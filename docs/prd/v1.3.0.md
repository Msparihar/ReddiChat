# PRD: ReddiChat v1.3.0 — Security & Cost Protection

**Status:** Draft
**Date:** 2026-03-28

## Version Goal

Secure the platform against abuse, cost runaway, and data corruption — making it safe for real public users. Every API request must be logged, every input must be bounded, and every user must have enforced daily limits.

## Current Security Gaps

1. **No request logging** — chat requests succeed silently. No visibility into who's using what, how much, or how often.
2. **Rate limits are per-hour only** — 30 requests/hour but no daily cap. A user can send 720 messages/day.
3. **No token/cost tracking** — a single message can be 5000 chars + images. No estimate of AI token cost. No daily token budget.
4. **File uploads barely validated** — only checks size (10MB). No MIME validation against actual content. No per-user daily upload limit. No max files per request.
5. **Conversation history unbounded** — sends up to 20 messages to AI with no token estimate. If messages are long, this blows up cost.
6. **No request body size limit** — FormData can be arbitrarily large.
7. **Error messages leak internals** — raw error.message from AI providers sent to client.

## Features

### 1. Comprehensive Request Logging
Log every API request with structured context:
- **Chat requests:** userId, model, message length (chars), file count, file total size, conversation history length, tools invoked, response length, duration, success/error
- **Reddit requests:** userId, username queried, endpoint, duration, success/error
- **Rate limit hits:** userId, endpoint, limit config, remaining
- Add `requestId` (UUID) to every request for tracing

**Files:** `frontend/app/api/chat/stream/route.ts`, all API routes, `frontend/lib/logger.ts`

### 2. Daily Usage Limits (Per User, Database-Backed)
Track daily usage in PostgreSQL so limits survive restarts and deploys:
- **New table:** `usage_tracking(id, user_id, date, message_count, estimated_tokens, upload_count, upload_bytes)` with `UNIQUE(user_id, date)`
- **Limits:** 100 messages/day, 500K estimated tokens/day, 50 uploads/day, 100MB upload size/day
- **Mechanism:** Upsert on every request: `INSERT ... ON CONFLICT (user_id, date) DO UPDATE SET message_count = message_count + 1, ...`
- **Check before processing:** Query current usage, reject if over limit
- Return clear error: "Daily limit reached. Resets at midnight UTC."
- Log every limit hit

**Files:** `frontend/lib/db/schema.ts` (new table), new `frontend/lib/usage.ts`, `frontend/app/api/chat/stream/route.ts`

### 3. Token Budget per Request
Before sending to AI, estimate the token cost:
- Estimate tokens: `Math.ceil(text.length / 4)` for each message
- Sum all history messages + current message
- Cap total input at ~32,000 estimated tokens (safe for all models)
- If over budget, truncate oldest history messages first
- Log the estimated token count per request

**Files:** `frontend/app/api/chat/stream/route.ts`

### 4. File Upload Hardening
- **Max files per request:** 5
- **Validate MIME type** — check file magic bytes match declared MIME type (at minimum: images, PDFs)
- **Allowed MIME types whitelist:** image/jpeg, image/png, image/gif, image/webp, application/pdf, text/plain, text/markdown
- **Per-user daily upload size tracking** — reject if cumulative exceeds 100MB
- **Sanitize filename** — strip path traversal chars, limit length to 200 chars
- Log every upload: userId, filename, size, MIME, accepted/rejected

**Files:** `frontend/app/api/chat/stream/route.ts`, `frontend/lib/validation.ts`, `frontend/lib/s3/index.ts`

### 5. Conversation History Token Budgeting
- Before building `aiMessages`, estimate total tokens of history
- If total exceeds budget (e.g., 24,000 tokens), drop oldest messages until under budget
- Always keep the system prompt + current message (reserve ~8,000 tokens for response)
- Log: history message count sent, estimated tokens, messages truncated

**Files:** `frontend/app/api/chat/stream/route.ts`

### 6. Sanitize Error Responses
- Never send raw `error.message` from AI providers to client
- Map known error types to user-friendly messages:
  - Rate limit → "The AI service is busy. Please try again in a moment."
  - Auth/key error → "Service configuration error. Please contact support."
  - Timeout → "Request timed out. Please try again."
  - Unknown → "Something went wrong. Please try again."
- Log the full error server-side, send sanitized version to client

**Files:** New `frontend/lib/error-handler.ts`, `frontend/app/api/chat/stream/route.ts`

### 7. Request Body Size Limit
- Add a maximum FormData size check before processing (e.g., 50MB total)
- Reject oversized requests early with 413 status

**Files:** `frontend/app/api/chat/stream/route.ts`

## Technical Scope

### New Files
- `frontend/lib/error-handler.ts` — error sanitization and mapping
- `frontend/lib/token-budget.ts` — token estimation and history truncation utilities

### Modified Files
- `frontend/lib/rate-limit.ts` — add daily tracking alongside hourly, add upload size tracking
- `frontend/lib/validation.ts` — add file validation schemas (MIME whitelist, filename sanitization, max files)
- `frontend/lib/logger.ts` — add requestId generation
- `frontend/app/api/chat/stream/route.ts` — comprehensive logging, daily limits, token budgeting, file hardening, error sanitization, body size check
- `frontend/app/api/chat/history/route.ts` — request logging
- `frontend/app/api/chat/history/[id]/route.ts` — request logging
- `frontend/app/api/reddit/user/[username]/route.ts` — request logging
- `frontend/app/api/reddit/user/[username]/posts/route.ts` — request logging
- `frontend/app/api/reddit/user/[username]/comments/route.ts` — request logging

### Database Changes
- New table: `usage_tracking` with unique constraint on `(user_id, date)`
- Requires `bun db:push` after schema change

### No New Dependencies
All changes use built-in Node.js APIs and existing Zod.

---

## Out of Scope

- Usage analytics dashboard (just tracking for now, UI in v2.1.0)
- WAF/CDN-level protection (Cloudflare rules)
- Prompt injection detection (complex, research-level problem)
- User blocking/banning system
- Admin dashboard for monitoring
