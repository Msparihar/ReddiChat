# PRD: ReddiChat v1.4.2 — Quick Topic Chips in Contact Dialog

**Status:** Draft
**Date:** 2026-03-28

## Version Goal

Add quick-select topic chips to the contact dialog so users can reach a sent message in 3 clicks: open → select topic → send.

---

## Problem

The contact dialog shipped in v1.4.1 is functional but requires manual effort for every contact attempt. Users must clear the pre-filled subject, figure out what to write, and type a message from scratch. For the most common contact reasons (upgrade request, billing question, bug report, feature request), this friction is unnecessary. A chip-first design eliminates the blank-slate problem: one tap pre-populates both the subject and a structured message template, leaving only the personal details to fill in.

---

## User Flow

```
Profile dropdown → "Contact for billing"
  → Dialog opens, chips visible at top
    → User clicks a chip (e.g. "Upgrade to Pro")
      → Subject + message auto-fill instantly
        → User edits message if needed → Send
          → toast.success → dialog closes
```

**Click count: 3** (open dialog → select chip → send). No typing required for the happy path.

---

## Features

### 1. Topic Chip Bar

A horizontal strip of pill-shaped filter chips rendered at the top of the dialog, above the existing form fields.

**Chips (in order):**

| Label | Key |
|---|---|
| Upgrade to Pro | `upgrade` |
| Billing Question | `billing` |
| Report Issue | `issue` |
| Feature Request | `feature` |
| Other | `other` |

**Visual design:**

- Shape: `rounded-full`
- Size: `px-3 py-1.5 text-xs font-medium`
- Minimum tap target on mobile: `min-h-[44px]` on the wrapping row, chips themselves at `py-1.5` (visually compact but the row provides touch area)
- Unselected state (light): `bg-gray-100 text-gray-600 hover:bg-gray-200 border border-transparent`
- Unselected state (dark): `bg-gray-800 text-gray-400 hover:bg-gray-700 border border-transparent`
- Selected state (both modes): `bg-purple-600 text-white border border-purple-600`
- Transition: `transition-colors duration-150`
- Layout: `flex flex-wrap gap-2` — chips wrap to a second row on narrow viewports
- No loading state — selection is instant (client-side only, no fetch)

**Placement in component tree:** immediately after `<DialogHeader>`, before the `space-y-4` form block, wrapped in its own `<div className="flex flex-wrap gap-2 pt-1 pb-2">`.

---

### 2. Prefill Templates

When a chip is selected, `subject` and `message` state are set synchronously via `setSubject` / `setMessage`. The `[auto-filled]` token in the "Upgrade to Pro" template is replaced with the actual `userEmail` prop at selection time (not at render time).

**Template table:**

| Chip | Subject value | Message value |
|---|---|---|
| Upgrade to Pro | `"Upgrade to Pro Plan"` | See below |
| Billing Question | `"Billing Question"` | See below |
| Report Issue | `"Bug Report"` | See below |
| Feature Request | `"Feature Request"` | See below |
| Other | `"General Inquiry"` | `""` (empty — user types from scratch) |

**Upgrade to Pro message:**
```
Hi, I'd like to upgrade my ReddiChat account to the Pro plan.

Current email: {userEmail}

Please let me know the next steps.
```

**Billing Question message:**
```
Hi, I have a question about billing:

[Type your question here]
```

**Report Issue message:**
```
Hi, I'd like to report an issue:

**What happened:**

**What I expected:**

**Steps to reproduce:**

```

**Feature Request message:**
```
Hi, I'd like to suggest a feature:

**Feature:**

**Why it would be useful:**

```

Templates are defined as a `const TOPIC_TEMPLATES` object at the top of `ContactDialog.tsx` — no external file, no fetch, no dynamic import.

---

### 3. Post-Selection Textarea Focus

After a chip is selected, the message textarea receives focus automatically so the user can immediately begin editing without a manual click.

Implementation: hold a `useRef<HTMLTextAreaElement>` on the textarea. Inside the chip `onClick` handler, call `ref.current?.focus()` after updating state. Place cursor at the end of the prefilled text using `ref.current.setSelectionRange(value.length, value.length)`.

---

### 4. Dialog Reset on Close

When the dialog closes (X button, Escape, backdrop), reset both `subject` and `message` to their defaults **and** clear `selectedChip` back to `null`. This ensures the dialog always opens in a clean, no-chip-selected state.

The existing `handleClose` function in `ContactDialog.tsx` (lines 68–74) is the right place to add `setSelectedChip(null)`.

---

### 5. State: Selected Chip Tracking

Add one new state variable:

```ts
const [selectedChip, setSelectedChip] = useState<string | null>(null);
```

`selectedChip` holds the key of the currently active chip (`"upgrade"`, `"billing"`, `"issue"`, `"feature"`, `"other"`) or `null` when none is selected. This drives the active/inactive visual styles on the chips.

When the user manually edits the subject or message textarea **after** selecting a chip, the chip remains highlighted (the active chip label tells the support team what the user's intent was — deselecting it on manual edit would be confusing and adds no value).

---

### 6. Dark Mode

All chip styles have explicit dark-mode variants via `isDark` conditional (the same `useTheme` pattern already used throughout `ContactDialog.tsx`). No Tailwind `dark:` prefix is needed — the component reads `isDark` from `useTheme()` and passes to `cn()` directly.

---

### 7. No New Dependencies

All changes are contained within `ContactDialog.tsx`. No new packages, no new files, no API changes.

## Implementation Notes

### `TOPIC_TEMPLATES` constant shape

```ts
type TopicTemplate = {
  label: string;
  subject: string;
  message: string; // empty string for "Other"
};

const TOPIC_TEMPLATES: Record<string, TopicTemplate> = {
  upgrade:  { label: "Upgrade to Pro",    subject: "Upgrade to Pro Plan",  message: "..." },
  billing:  { label: "Billing Question",  subject: "Billing Question",     message: "..." },
  issue:    { label: "Report Issue",      subject: "Bug Report",           message: "..." },
  feature:  { label: "Feature Request",   subject: "Feature Request",      message: "..." },
  other:    { label: "Other",             subject: "General Inquiry",      message: "" },
};
```

The "Upgrade to Pro" message template stores `{userEmail}` as a literal placeholder and the click handler replaces it with the actual `userEmail` prop at selection time:

```ts
message: template.message.replace("{userEmail}", userEmail)
```

### Chip click handler

```ts
const handleChipSelect = (key: string) => {
  const t = TOPIC_TEMPLATES[key];
  setSelectedChip(key);
  setSubject(t.subject);
  setMessage(key === "upgrade" ? t.message.replace("{userEmail}", userEmail) : t.message);
  // focus textarea after state flush
  setTimeout(() => {
    if (textareaRef.current) {
      textareaRef.current.focus();
      const len = textareaRef.current.value.length;
      textareaRef.current.setSelectionRange(len, len);
    }
  }, 0);
};
```

`setTimeout(..., 0)` defers the focus call until after React has flushed the state update and re-rendered the textarea with its new value — necessary because `setSelectionRange` must see the updated value.

### Chip bar JSX skeleton

```tsx
<div className="flex flex-wrap gap-2 pt-1 pb-2">
  {Object.entries(TOPIC_TEMPLATES).map(([key, t]) => (
    <button
      key={key}
      type="button"
      onClick={() => handleChipSelect(key)}
      className={cn(
        "rounded-full px-3 py-1.5 text-xs font-medium transition-colors duration-150",
        selectedChip === key
          ? "bg-purple-600 text-white border border-purple-600"
          : isDark
            ? "bg-gray-800 text-gray-400 hover:bg-gray-700 border border-transparent"
            : "bg-gray-100 text-gray-600 hover:bg-gray-200 border border-transparent"
      )}
    >
      {t.label}
    </button>
  ))}
</div>
```

### `sm:max-w-md` is sufficient

Five short chip labels fit on a single row at `sm:max-w-md` (448 px) on desktop. On mobile (< 448 px) they wrap to two rows via `flex-wrap` — tested visually at 375 px:

- Row 1: Upgrade to Pro · Billing Question · Report Issue
- Row 2: Feature Request · Other

This is acceptable. No horizontal scroll is introduced.

---

## Out of Scope

- Animated chip-to-chip transition (content fade/crossfade between templates) — the instant swap is intentional and feels snappy; an animation would add perceived lag
- Server-side template rendering or dynamic templates fetched from an API
- Allowing users to save custom templates
- A "clear selection" button to deselect the active chip (editing the textarea fields serves this purpose implicitly)
- Any changes to `ProfileDropdown.tsx`
- Any changes to `POST /api/contact`
- New Tailwind config values — all classes used are standard Tailwind utilities already in the project
