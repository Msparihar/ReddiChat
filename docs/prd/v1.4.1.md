# PRD: ReddiChat v1.4.1 — In-App Billing Contact Form

**Status:** Draft
**Date:** 2026-03-28

## Version Goal

Replace the `mailto:` link in the profile dropdown with an in-app contact dialog so users can send billing inquiries without leaving the browser.

---

## Problem

v1.4.0 added a "Contact for billing" item to the profile dropdown (`frontend/components/layout/ProfileDropdown.tsx`, line 266). It is implemented as a raw `<a href="mailto:manishsparihar2020@gmail.com">` link. Clicking it hands off to whatever email client the OS has configured — or, on many machines, does nothing at all. This is broken UX for a web product: users expect in-app flows, not OS handoffs. This patch replaces the link with a modal form backed by a proper API endpoint.

---

## Features

### 1. `Dialog` shadcn wrapper component

`@radix-ui/react-dialog` is already installed (confirmed in `frontend/package.json`) but no shadcn wrapper exists in `frontend/components/ui/`. A standard shadcn `dialog.tsx` wrapper must be created so the rest of the codebase can consume `<Dialog>`, `<DialogContent>`, `<DialogHeader>`, `<DialogTitle>`, `<DialogDescription>`, and `<DialogFooter>` from `@/components/ui/dialog`.

**New file:** `frontend/components/ui/dialog.tsx`

The wrapper follows the same shadcn pattern already used by `frontend/components/ui/dropdown-menu.tsx` — thin re-exports from the Radix primitive with Tailwind class names applied via `cn()`.

---

### 2. `ContactDialog` component

A self-contained modal with a contact form. Exported from a new file and rendered inside `ProfileDropdown`.

**New file:** `frontend/components/layout/ContactDialog.tsx`

**Props:**

```ts
interface ContactDialogProps {
  open: boolean;
  onOpenChange: (open: boolean) => void;
  userEmail: string;
}
```

**Form fields:**

| Field | Type | Default value | Editable |
|---|---|---|---|
| Email | `<input type="email">` | Session user email | No (read-only) |
| Subject | `<input type="text">` | `"Billing Inquiry — ReddiChat"` | Yes |
| Message | `<textarea>` | Empty | Yes |

Reuses `frontend/components/ui/input.tsx` and `frontend/components/ui/textarea.tsx` (both already exist). Uses `frontend/components/ui/button.tsx` for the Send button.

**Behavior:**

- Form submits via `fetch('POST /api/contact', { credentials: 'include' })` with JSON body `{ subject, message }` (email is inferred server-side from the session — do not send it in the body).
- Send button shows a loading state (`disabled` + spinner or "Sending…" text) while the request is in flight.
- On `200`: fire `toast.success("Message sent!")` via `react-hot-toast` (already a project dependency), then close the dialog and reset the form.
- On any non-200 or network error: fire `toast.error("Failed to send message. Please try again.")`.
- Closing the dialog (X button, Escape key, or backdrop click) resets the form state.
- Subject field: max 200 characters. Message field: max 2000 characters, show a character counter (`X / 2000`).

### 4. `POST /api/contact` endpoint

**New file:** `frontend/app/api/contact/route.ts`

**Authentication:** Reads the session via Better Auth's `auth.api.getSession`. Returns `401` if no valid session exists.

**Rate limiting:** Add a new config to `RATE_LIMITS` in `frontend/lib/rate-limit.ts`:
```ts
contact: { maxRequests: 5, windowMs: 86400000 } as RateLimitConfig, // 5/day
```
Use `checkRateLimit('contact:${userId}', RATE_LIMITS.contact)`. On limit exceeded, return `429` with `{ error: "Too many requests. You can send 5 contact messages per day." }`.

**Request body** (validated with `zod`):

```ts
const ContactSchema = z.object({
  subject: z.string().min(1).max(200),
  message: z.string().min(1).max(2000),
});
```

Return `400` with `{ error: "Invalid request" }` if validation fails.

**Email sending:** Calls a new exported function `sendContactEmail` from `frontend/lib/email/index.ts` (see Feature 5). Does not instantiate a new transporter — reuses the existing one in that file.

**Success response:** `200` with `{ ok: true }`.

**Error handling:** If `sendContactEmail` throws, log the error and return `500` with `{ error: "Failed to send message" }`.

---

### 5. `sendContactEmail` in the email service

**Modified file:** `frontend/lib/email/index.ts`

Add a new exported async function alongside the existing `sendLoginEmail`:

```ts
export async function sendContactEmail(
  fromUserEmail: string,
  subject: string,
  message: string
): Promise<void>
```

- Uses the existing `transporter` — no new `nodemailer.createTransport` call.
- Guards on `SMTP_USER` / `SMTP_PASSWORD` being set, same as `sendLoginEmail`.
- Sends to `manishsparihar2020@gmail.com` (hardcoded owner address).
- `from` field: `"ReddiChat" <${process.env.SMTP_USER}>`.
- `replyTo` field: set to `fromUserEmail` so the owner can reply directly.
- `subject` field: the user-provided subject (already validated to max 200 chars).
- `html` body: a simple styled layout consistent with `sendLoginEmail` — show the sender's email address, their subject, and their message. **HTML-escape** the subject and message content before inserting into the template to prevent email injection (use a simple replace for `<`, `>`, `&`, `"`).

## Technical Scope

### New Files

| File | Purpose |
|---|---|
| `frontend/components/ui/dialog.tsx` | Install via `npx shadcn@latest add dialog` (Radix already installed) |
| `frontend/components/layout/ContactDialog.tsx` | Contact form modal component |
| `frontend/app/api/contact/route.ts` | `POST /api/contact` — auth-gated, rate-limited email send |

### Modified Files

| File | What changes |
|---|---|
| `frontend/components/layout/ProfileDropdown.tsx` | Replace `mailto:` anchor with `ContactDialog` trigger |
| `frontend/lib/email/index.ts` | Add `sendContactEmail` function reusing existing transporter |
| `frontend/lib/rate-limit.ts` | Add `contact` rate limit config (5/day) |

### New Dependencies

- None. `@radix-ui/react-dialog`, `nodemailer`, `zod`, and `react-hot-toast` are all already installed.

### Database Changes

- None — rate limiting uses the existing mechanism already in place for other API routes (no new table required).

- No new env vars. The existing `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, and `SMTP_PASSWORD` used by `sendLoginEmail` are sufficient.
