# PRD: ReddiChat v1.3.1 â€” Usage Visibility

**Status:** Reviewed
**Date:** 2026-03-28

## Version Goal

Surface daily usage data to users so they understand their remaining quota before hitting a limit, and replace generic limit-hit errors with precise, actionable messages.

---

## Problem

v1.3.0 shipped database-backed daily limits (`usage_tracking` table, `frontend/lib/usage.ts`) but built zero UI around them. The result: users send a message, receive a vague error like "Daily message limit reached. Resets at midnight UTC.", have no idea how close they were, and have no visibility into their usage between messages. This patch closes that gap.

---

## Features

### 1. `GET /api/usage` Endpoint

A new authenticated JSON endpoint that returns the authenticated user's daily usage for today, the configured limits, and the UTC reset time.

**Response shape:**

```json
{
  "usage": {
    "messages": 43,
    "tokens": 187400,
    "uploads": 7,
    "uploadBytes": 10485760
  },
  "limits": {
    "messages": 100,
    "tokens": 500000,
    "uploads": 50,
    "uploadBytes": 104857600
  },
  "resetAt": "2026-03-29T00:00:00.000Z"
}
```

- Returns 401 if unauthenticated.
- `resetAt` is always midnight UTC of the **following** day: `new Date(new Date(getTodayDate() + "T00:00:00Z").getTime() + 86400000).toISOString()`
- Calls the existing `getDailyUsage(userId)` from `frontend/lib/usage.ts` â€” no new DB queries.
- Imports `DEFAULT_LIMITS` from `frontend/lib/usage.ts` and transforms keys for the response:
  - `maxMessagesPerDay` â†’ `messages`
  - `maxTokensPerDay` â†’ `tokens`
  - `maxUploadsPerDay` â†’ `uploads`
  - `maxUploadBytesPerDay` â†’ `uploadBytes`
- Rate limited using existing `RATE_LIMITS.reddit` (60/hr) to prevent polling abuse.

**New file:** `frontend/app/api/usage/route.ts`

### 2. Usage Indicator in `MessageInput`

A compact, always-visible usage line rendered below the model selector inside the bottom bar of `MessageInput`, showing remaining messages for today.

**Display rules:**

- Fetch `/api/usage` with React Query (`useQuery`) on mount and after each sent message (invalidate on `sendMessage` completion).
- Show: `"X messages left today"` where X is `limits.messages - usage.messages`.
- Color thresholds:
  - `> 20` remaining: muted gray (`text-gray-400` / `text-gray-500` dark)
  - `6â€“20` remaining: amber (`text-amber-500`)
  - `1â€“5` remaining: red (`text-red-500`)
  - `0` remaining: red + bold + disable send button
- On fetch error or loading: show "â€”" placeholder (do not block the input).
- Tooltip on hover: full breakdown â€” "Messages: X/100 | Tokens: X/500,000 | Uploads: X/50" (use `toLocaleString()` for numbers)

**Modified file:** `frontend/components/chat/MessageInput.tsx`

**New hook file:** `frontend/hooks/use-usage.ts`
- Wraps `useQuery({ queryKey: ['usage'], queryFn: () => fetch('/api/usage', { credentials: 'include' }).then(r => r.json()), refetchInterval: 60000 })`
- Exports `useUsage()` returning `{ data, isLoading, isError }`
- Also exports `useInvalidateUsage()` which returns a function using `useQueryClient().invalidateQueries({ queryKey: ['usage'] })`
- Invalidation should be called in `chat-store.ts` after the `done` SSE event is processed (in the `finishStreaming` path)

### 3. Precise Limit-Hit Error Messages

When a limit check fails in `frontend/app/api/chat/stream/route.ts`, the SSE error payload currently sends `dailyCheck.reason` which is a static string like "Daily message limit reached. Resets at midnight UTC." It does not include actual counts.

Update `checkDailyLimit` in `frontend/lib/usage.ts` to return richer `reason` strings that include exact counts:

| Limit hit | New `reason` string |
|---|---|
| Message count | `"You've used 100/100 messages today. Resets at midnight UTC."` |
| Token count | `"You've used 500,000/500,000 tokens today. Resets at midnight UTC."` |
| Upload count | `"You've used 50/50 uploads today. Resets at midnight UTC."` |
| Upload bytes | `"You've used 100MB/100MB of daily upload quota. Resets at midnight UTC."` |

The `reason` field already flows through `route.ts` â†’ SSE `error` event â†’ `chat-store.ts` `handleStreamError` â†’ rendered as an error message bubble in the chat. No changes to the store or stream handler are needed for this feature; the fix is entirely in `frontend/lib/usage.ts`.

**Formatting rules:**
- Token counts: `toLocaleString()` (e.g., "500,000")
- Upload bytes: divide by `1024 * 1024`, round to nearest integer, append "MB" (e.g., "100MB")
- Add a helper `formatBytes(bytes: number): string` in `usage.ts` for byte formatting

**Modified file:** `frontend/lib/usage.ts`

## Out of Scope

- Full usage dashboard or history page (planned for v2.1.0 per `docs/prd/v2.1.0.md`)
- Token or upload quota indicators (messages remaining is the most actionable signal; others visible in tooltip only)
- Admin-level usage monitoring
- Per-model token limit display
- Proactive warnings mid-conversation (e.g., "You have 5 messages left" toast)
- Changing limit values or making limits configurable per user
