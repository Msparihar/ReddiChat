# PRD: ReddiChat v1.6.0 — User Roles & Tier-Based Access Control

**Status:** Draft
**Date:** 2026-03-28

## Version Goal

Build the foundational roles and tier-based limits system so that v2.1.0 (Stripe billing) can add payments on top without touching the enforcement logic.

## Background & Motivation

Today every user shares identical hardcoded limits (`DEFAULT_LIMITS` in `frontend/lib/usage.ts`: 100 msgs/day, 500K tokens/day, 50 uploads/day, 100MB/day). The model selector in `MessageInput` shows every model to every user with no access control. The plan badge in `Sidebar` is hardcoded to the string `"Free"`. There are no admin tools to inspect or adjust users.

v2.1.0 (Monetization) depends on tiers already existing. This version builds exactly that — no Stripe, no billing UI, just the data model, the config, the enforcement, and the admin API.

---

## Features

### 1. User Role Column
Add a `role` field to the `user` table with four possible values: `"free"`, `"pro"`, `"team"`, `"admin"`. All new accounts default to `"free"`. The owner account is set to `"admin"` via a one-time migration script.

Better Auth surfaces session data through its Drizzle adapter by reading the `user` table directly. Adding a column to the table makes the value available in `session.user` if Better Auth is configured to expose custom fields via `additionalFields`. This version adds that config.

### 2. Tier Limits Configuration (Single Source of Truth)
A new file `frontend/lib/tiers.ts` defines all per-tier limits and access rules. Nothing else hardcodes a limit — every consumer imports from this file. This is what v2.1.0 will read to display pricing tables.

Limits per tier:

| Limit | Free | Pro | Team | Admin |
|---|---|---|---|---|
| Messages/day | 100 | 1,000 | 5,000 | `Infinity` |
| Tokens/day | 500,000 | 5,000,000 | 20,000,000 | `Infinity` |
| Uploads/day | 50 | 200 | 500 | `Infinity` |
| Upload bytes/day | 100 MB | 1 GB | 5 GB | `Infinity` |
| Allowed model IDs | `["gemini-2.5-flash"]` | all | all | all |
| Tools | `["reddit", "web"]` | all | all | all |

`Infinity` limits are treated as "no limit" — the check is skipped, not compared to a huge number.

### 3. Tier-Aware Enforcement

**`frontend/lib/usage.ts`**
- Add `getTierLimits(role: UserRole): UsageLimits` — reads from `tiers.ts`, returns the correct `UsageLimits` object.
- `checkDailyLimit` already accepts a `limits` parameter; callers currently omit it, falling back to `DEFAULT_LIMITS`. Change those call sites to pass `getTierLimits(role)` instead. `DEFAULT_LIMITS` is kept as an alias for `getTierLimits("free")` for backward compatibility.

**`frontend/app/api/chat/stream/route.ts`**
- After the session check, fetch `session.user.role` (defaulting to `"free"` if absent for existing sessions).
- Pass `getTierLimits(role)` to both `checkDailyLimit("message")` and `checkDailyLimit("upload")` calls.
- After input validation but before the AI call, check whether `modelId` is in the tier's `allowedModels`. If not, return a 403 SSE error: `"Your plan does not include access to this model. Upgrade to Pro to unlock all models."`
- Pass the role to `streamChatResponse` (or filter tool availability there) so tool access can be gated in `frontend/lib/ai/agent.ts`.

**`frontend/app/api/usage/route.ts`**
- Replace `DEFAULT_LIMITS` with `getTierLimits(session.user.role)`.
- Add `role` and `tier` fields to the response so the frontend can display the correct plan name and know which models/tools are available without a separate request.

### 4. Admin API Endpoints

Both endpoints require `session.user.role === "admin"`. Non-admin requests receive a 403 JSON response. Both endpoints also apply rate limiting using the existing `checkRateLimit` utility (`20 requests/hour` per admin user ID).

**`GET /api/admin/users`**
Returns a paginated list of all users with their role and today's usage. Supports `?page=1&limit=50` query params.

Response shape:
```json
{
  "users": [
    {
      "id": "...",
      "name": "...",
      "email": "...",
      "role": "free",
      "createdAt": "...",
      "usage": {
        "messages": 12,
        "tokens": 45000,
        "uploads": 2,
        "uploadBytes": 1048576
      }
    }
  ],
  "total": 143,
  "page": 1,
  "limit": 50
}
```

**`PATCH /api/admin/users/[id]/role`**
Changes a user's role. Body: `{ "role": "pro" | "free" | "team" | "admin" }`. Validated with Zod. Returns the updated user. Logs the role change with `requestId`, `adminId`, `targetUserId`, `oldRole`, `newRole`.

### 5. UI Changes

**`frontend/components/layout/Sidebar.tsx`**
- The plan badge under the user's name currently hardcodes `"Free"`. Replace it with the actual role from `session.user.role`.
- Display mapping: `free` → "Free", `pro` → "Pro", `team` → "Team", `admin` → "Admin".
- For `free` role: show an "Upgrade" link next to the badge that navigates to `/pricing` (page does not need to exist yet — link is enough as a placeholder for v2.1.0).
- Admin role gets a distinct badge color (e.g., amber) to make it visually obvious.

**`frontend/hooks/use-usage.ts`** and **`frontend/components/chat/MessageInput.tsx`**
- `use-usage.ts`: extend `UsageData` type to include `role` and `tier` fields returned by the updated `/api/usage`.
- `MessageInput`: when `messagesRemaining` drops to 20% or less of the tier limit (not a hardcoded 20), show the amber warning. When `messagesRemaining <= 5`, show a red "Upgrade to Pro" prompt as a small inline banner above the input box (free tier only). The banner includes a link to `/pricing`.
- Model selector dropdown: for each model in `AI_MODELS`, check whether the model's `id` is in the tier's `allowedModels`. Unavailable models render with `opacity-50`, a `cursor-not-allowed` style, a lock icon, and a tooltip `"Available on Pro and above"`. Clicking a locked model does nothing (`onSelect` is suppressed). If the user's currently selected model is not in their tier (e.g., after an admin downgrades them), silently reset `selectedModel` to `DEFAULT_MODEL_ID` on mount via a `useEffect` in `MessageInput`.

**`frontend/stores/chat-store.ts`**
- No structural changes. The model validation on mount is handled in `MessageInput` rather than the store to keep store logic clean.

### Database Changes

One migration:
```sql
-- Add role column to user table
ALTER TABLE "user"
  ADD COLUMN IF NOT EXISTS role VARCHAR(20) NOT NULL DEFAULT 'free';

-- Set owner account to admin (replace with actual owner user ID)
UPDATE "user"
  SET role = 'admin'
  WHERE email = '<OWNER_EMAIL>';
```

The migration file goes in `frontend/lib/db/migrations/`. Run via `drizzle-kit migrate`.

### Better Auth Configuration

Better Auth requires explicit registration of custom fields on the user model to include them in the session token. In `frontend/lib/auth/index.ts`, add:

```ts
user: {
  additionalFields: {
    role: {
      type: "string",
      defaultValue: "free",
      fieldName: "role",
    },
  },
},
```

This makes `session.user.role` available in all server contexts without a separate DB lookup.

### New Dependencies

None. All changes use existing dependencies (Drizzle, Better Auth, Zod, React Query, Zustand).

## Out of Scope

- Stripe integration, subscription management, or billing UI — that is v2.1.0
- Admin UI (user management dashboard) — API endpoints only in this version
- `/pricing` page content — the link is added in the UI but the page itself is v2.1.0
- Team-specific features (multi-seat, shared workspace, API keys) — v2.1.0
- Tool access enforcement in the agent beyond the basic `allowedTools` filter (e.g., full B2B API) — v2.1.0
- Email notifications on role change
- Role-based audit log table
